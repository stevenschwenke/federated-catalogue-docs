== Architecture Decisions

=== ADR 1: Monolithic vs. Microservices
**Context:** Are the subcomponents implemented as separate micro-services or are they integrated into a single service.

1. **Microservice**: Usage of different technologies possible, independent development, scalability, ...
2. **Single Service**: Easier maintainability, understanding, easier to handle concurrency (especially conflicting operations like adding an asset and changing schema)

**Decision:** Implement catalogue as a modular, single service. It's easier to integrate the different components. Most components are called by only one other component. A distributed microservice implementation only introduces additional communication overhead. Scalability issues are not expected, since the main load is read-only and time consuming operations are queries against the graph db, which is a separate component that can be horizontally scaled.
Therefore, the Federated Catalogue is considered as microservice application within the GXFS ecosystem. The federated catalogue itself also consists of different, individually deployable components.

=== ADR 2: Intra Catalogue communication
**Context:** If the catalogue is implemented as one component, the subcomponents can communication using Method-calls. Another option is to implement an internal event-/notification system, whereby each component registers itself as a listener on other components or a central message bus.

1. **Method-Calls:** Easy to implement, clear structure and easy to debug. Limited options for extendability.
2. **Event-based:** Highly plugable systems, where new components can be added without other components knowing that. Additional indirection which causes complexity. Harder to debug.

**Decision:** The Catalogue es a static, pre-specified and configured component. Adding plugins during runtime is not needed. The maintainability of the catalogue is more important than additional flexibility.

=== ADR 3: Disable Signature Verification by Default

**Context:** The verification pipeline has four independent flags: `semantics`, `schema`, `vp-signature`, and `vc-signature`. Signature verification resolves the `verificationMethod` DID, retrieves the public key, and verifies the JWS proof. However, without the Gaia-X trust framework (`gaiax.enabled=true`), the `x5u` certificate chain that binds the DID to a vetted legal entity is never validated.

Without the trust anchor chain, signature verification only proves internal consistency (the credential was signed by whoever controls that DID) and tamper detection (the payload has not been modified). It does not establish identity, trust, or accountability â€” anyone can create a `did:web` on a server they control and sign arbitrary claims.

This makes signature verification without trust anchor enforcement ineffective as a security measure. Meanwhile, keeping signatures enabled by default forces all deployments to stand up DID resolution infrastructure even when not using the Gaia-X trust framework.

**Decision:** Disable `vp-signature` and `vc-signature` in the default application profile (`application.yml`). Signature verification is re-enabled in the strict deployment profile alongside the full trust framework (Gaia-X enabled, schema validation, trust anchor registry).

[cols="1,1,1"]
|===
| Flag | Default profile | Strict profile

| `semantics` | `true` | `true`
| `schema` | `false` | `true`
| `vp-signature` | `false` | `true`
| `vc-signature` | `false` | `true`
| `gaiax.enabled` | `false` | `true`
|===

This aligns with CAT-FR-SF-04 (remove forced validation): validation flags that provide no security value without the trust framework should not be forced by default. Production deployments must explicitly enable signatures together with the trust framework.

=== ADR 4: Non-RDF Asset Storage Strategy

**Context:** The catalogue needs to support non-RDF assets (PDFs, plain JSON, YAML, images, binary files) alongside RDF credentials. Several approaches were considered:

1. **Separate `/assets` endpoint and storage:** New endpoint, new database table, new service layer. Clean separation but duplicates lifecycle management, filtering, and metadata handling.
2. **Unified endpoint with dual-path routing:** Extend the existing `POST /assets` endpoint (formerly `POST /self-descriptions`) to accept non-RDF content. Reuse the `assets` table (formerly `sdfiles`, extended with asset metadata columns), the `AssetStore` interface, and the existing filter/lifecycle infrastructure. Route to verification or direct storage based on content-type detection.
3. **External object storage (S3/MinIO):** Delegate non-RDF storage to an external system. Adds infrastructure complexity and a new dependency.

**Decision:** Option 2 â€” unified endpoint with dual-path routing.

* The `POST /assets` endpoint accepts `multipart/form-data` and `application/octet-stream` in addition to `application/json` and `application/ld+json`.
* An `RdfDetector` classifies uploads by content type (configurable allowlist) and content inspection (JSON-LD `@context` pattern).
* RDF content follows the existing verification + graph storage path.
* Non-RDF content bypasses verification and graph storage entirely. It is stored in a dedicated `assetFileStore` (separate directory from schema/context caches) with metadata in the `assets` table.
* Three columns are added to `assets`: `content_type` (VARCHAR), `file_size` (BIGINT), `original_filename` (VARCHAR). The `content` column is made nullable (non-RDF assets store content only in the File Store).
* Non-RDF assets receive auto-generated IRI identifiers: UUID URNs (`urn:uuid:...`) per RFC 4122, assigned by `IriGenerator` (see ADR 7).

**Rationale:** Reusing the existing endpoint and metadata infrastructure avoids duplication of lifecycle management, filtering, pagination, and access control. The `RdfDetector` allowlist gives operators explicit control over which content types trigger verification. Separate File Store instances keep asset blobs isolated from schema caches for operational clarity.

=== ADR 5: Fine-Grained Permission-Based Authorization

**Context:** The original authorization model used four coarse-grained roles (Ro-MU-CA, Ro-MU-A, Ro-AS-Afootnote:[Originally named `Ro-SD-A` (Self-Description Administrator). Renamed to `Ro-AS-A` (Asset Administrator) to align with the generic asset terminology.], Ro-PA-A) to control API access. This presented several limitations:

1. **No read-level control.** Asset and schema read endpoints were open to any authenticated user, with no way to restrict read access.
2. **No separation of concerns.** The Ro-AS-A and Ro-MU-A roles granted full write access to assets, with no way to differentiate between create, update (revoke), and delete operations.
3. **No schema write control.** Schema management was restricted only to Ro-MU-CA, with no intermediate access level for schema administrators.
4. **No query access control.** The query endpoint was accessible to any authenticated user.

Two approaches were considered:

1. **Extend existing roles:** Add more coarse-grained roles (e.g., Ro-Schema-A). Simple but doesn't address the granularity problem and leads to role proliferation.
2. **Fine-grained permissions with composite roles:** Define atomic permissions (ASSET_CREATE, ASSET_READ, etc.) and compose them into roles. Maximum flexibility while maintaining backward compatibility.

**Decision:** Option 2 â€” fine-grained permission-based authorization.

Ten atomic permissions are defined: `ASSET_CREATE`, `ASSET_READ`, `ASSET_UPDATE`, `ASSET_DELETE`, `SCHEMA_CREATE`, `SCHEMA_READ`, `SCHEMA_UPDATE`, `SCHEMA_DELETE`, `QUERY_EXECUTE`, and `ADMIN_ALL`. Each API endpoint requires a specific permission (or `ADMIN_ALL`). Permissions are implemented as Keycloak client roles and flow into Spring Security authorities via the existing `CustomJwtAuthenticationConverter`.

Backward compatibility is maintained by converting the legacy roles (Ro-MU-CA, Ro-MU-A, Ro-AS-A) into Keycloak composite roles that include the appropriate permissions. Users with legacy roles automatically receive the corresponding fine-grained permissions without any migration.

**Rationale:** Fine-grained permissions enable precise access control for multi-tenant deployments where different user groups need different levels of access to catalogue resources. The composite role mechanism ensures zero-disruption migration for existing deployments. No changes to JWT extraction or authentication infrastructure were required.

=== ADR 6: Claim Class Naming Convention

**Context:** CAT-NFR-01footnote:cat-enhancement[Enhancement of the XFSC Federated Catalogue. 2025. https://github.com/eclipse-xfsc/docs/blob/9a178c30f76610e5924fd72c39d259d6a8ae0461/federated-catalogue/src/docs/CAT%20Enhancement/CAT_Enhancement_Specifications%20v1.0.pdf] renamed the legacy `SdClaim` class (tied to Self-Description terminology). The team needed to decide on the new name. Three options were considered:

1. **`Claim`** â€” generic, forward-looking, but too broad for the current implementation which only extracts claims from credentials.
2. **`AssetClaim`** â€” aligns with the generic asset terminology but is misleading because non-RDF assets cannot have claims.
3. **`CredentialClaim`** â€” accurate for the current scope (only credentials produce claims), with a planned rename when scope expands.

The discussion established the following asset taxonomy:

[mermaid]
....
classDiagram
    class Asset
    class NonRDFAsset
    class RDFAsset
    class Credential
    class NonCredential
    class Claim {
        &lt;&lt;concept&gt;&gt;
    }

    Asset <|-- RDFAsset
    Asset <|-- NonRDFAsset
    RDFAsset <|-- Credential
    RDFAsset <|-- NonCredential
    Credential ..> Claim : contains
    NonCredential ..> Claim : contains

    note for Claim "A Claim is the atomic unit\nof semantic assertion (Triple)."
....

Key concepts:

* Non-RDF assets cannot have claims â€” they are opaque binary content.
* RDF assets can be credentials (verified VPs/VCs) or non-credentials (plain Turtle, RDF/XML).
* All RDF assets contain triples, and claims are synonymous with triples in this context.
* Even non-credential RDF assets contain claims; they are simply unverified.

**Decision:** Option 3 â€” `CredentialClaim`, with a phased naming strategy:

[cols="1,2,2",options="header"]
|===
| Phase | Class Name | Rationale
| Legacy | `SdClaim` | Original name tied to Self-Description terminology.
| Current (CAT-NFR-01) | `CredentialClaim` | Only credentials produce claims in the current codebase.
| Future (CAT-FR-GD-02) | `RdfClaim` | When non-credential RDF claim extraction is implemented (story 008), claims can originate from any RDF asset.
|===

**Rationale:** Naming for the current scope avoids premature abstraction. The rename to `RdfClaim` is deferred to the story (CAT-FR-GD-02) that actually introduces non-credential claim extraction, following a "name for what it is now, rename when scope expands" principle. This keeps each rename tied to a concrete implementation change rather than a speculative future.

=== ADR 7: IRI-Based Asset Identification

**Context:** CAT-FR-AM-02: Each Asset MUST be identified by an IRI. DIDs and UUIDs as URNs (RFC 4122) MUST be supported.footnote:cat-enhancement[] The previous implementation used content-hash-based identifiers (`urn:asset:sha256:<hash>`) for non-RDF assets and `credentialSubject` for RDF credentials. Three approaches were considered:

1. **Keep hash-based identifiers:** Simple but violates CAT-FR-AM-02 â€” content hashes are not IRIs in the W3C sense, and they couple identity to content (any content change produces a new identifier).
2. **UUID URNs for all assets:** Assign a UUID URN (`urn:uuid:...`) per RFC 4122footnote:[https://www.rfc-editor.org/rfc/rfc4122[RFC 4122 â€” A Universally Unique IDentifier (UUID) URN Namespace]] to every asset regardless of type. Simple and uniform but discards the meaningful identifiers already present in RDF content (DIDs, ontology IRIs).
3. **IRI extraction with UUID fallback:** Extract IRIs from RDF content where possible (`credentialSubject.id`, `@id`, ontology/shapes/vocabulary IRIs), generate UUID URNs for non-RDF assets and as fallback for RDF content without an extractable IRI.

**Decision:** Option 3 â€” IRI extraction with UUID URN fallback.

Two new components are introduced:

* **`IriGenerator`** â€” extracts IRIs from RDF content (Verifiable Credentials via `credentialSubject.id`, generic JSON-LD via `@id`, OWL ontologies, SHACL shapes, SKOS concept schemes) or generates UUID URNs per RFC 4122 for non-RDF assets and as fallback.
* **`IriValidator`** â€” checks IRI formats via `isValid(iri): boolean`. Supported formats: DIDs (`did:<method>:<id>`), UUID URNs (`urn:uuid:<uuid>`), generic URNs (`urn:<nid>:<nss>`), and HTTP/HTTPS IRIs. Returns true/false â€” callers decide how to handle invalid IRIs.

The public API is extended with IRI-based methods (`getById`, `getFileById`) alongside the existing hash-based methods. Read endpoints use `/assets/{id}` where `{id}` is the URL-encoded IRI. Mutation operations (delete, revoke) remain hash-based â€” see the Hash-Based Mutation Operations Exception below.

The `federated-catalogue.asset.subject-id-prefix` configuration property is removed â€” IRI generation is now handled entirely by `IriGenerator`.

**Rationale:** Extracting meaningful IRIs from RDF content preserves semantic identity (a DID or ontology IRI carries provenance information that a UUID does not). UUID URNs for non-RDF assets satisfy RFC 4122 and CAT-FR-AM-02 while providing unique, stable identifiers without coupling identity to content. The content hash is retained internally for deduplication and file storage but is no longer the public identifier.

==== Hash-Based Mutation Operations Exception

The delete and revoke operations are the deliberate exceptions to IRI-based identification â€” both retain hash-based targeting:

* `DELETE /assets/{asset_hash}` â€” permanently removes an asset
* `POST /assets/{asset_hash}/revoke` â€” changes lifecycle status to REVOKED

The same rationale applies to both operations:

1. **Data model constraint** The `subjectid` column (the IRI) is not globally unique. The partial unique index `idx_asset_active` enforces uniqueness only among rows with `status=0` (ACTIVE). Multiple rows can share the same `subjectid` when they have different statuses (DEPRECATED, REVOKED, EOL). Mutation via `WHERE subjectid = :id` could match multiple rows â€” this is unacceptable for destructive or state-changing operations.

2. **Mutation semantics** Delete and revoke are irreversible operations that demand unambiguous single-row targeting. The `asset_hash` column is the PRIMARY KEY â€” it guarantees exactly one row is affected. The IRI identifies a _logical asset_; the hash identifies a _specific content artifact_. Mutations operate at the physical artifact level, not the logical level. For revoke specifically, hash-based lookup finds the asset regardless of its current status, enabling the business logic to return a proper 409 Conflict (e.g., attempting to revoke an already deprecated asset) rather than a misleading 404.

3. **Versioning readiness (CAT-FR-LM-01)footnote:cat-enhancement[]** The planned version control feature will introduce multiple version rows per `subjectid`, each with a different `asset_hash` and status. IRI-based mutation would become increasingly ambiguous â€” which version should be affected? Hash-based targeting remains precise regardless of how many versions exist. When versioning is introduced, both endpoints will be replaced by version-specific operations (`POST /assets/{id}/versions/{v}/revoke`).

4. **Audit versioning forward-look** When audit history tracking is introduced (e.g., Hibernate Envers), physical deletion conflicts with audit trail immutability. At that point, the delete operation should transition to a lifecycle status change (e.g., â†’ EOL) rather than physical row removal. Retaining hash-based targeting keeps the migration path clean: `DELETE WHERE hash` transitions to `UPDATE status WHERE hash`. The targeting mechanism (hash) stays consistent across this evolution.
