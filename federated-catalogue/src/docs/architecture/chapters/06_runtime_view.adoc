== Runtime View

This chapter first defines a generic workflow for write operations (<<_generic_control_flow_for_write_operations>>). It's followed by the scenarios that implement the API endpoints (<<_list_of_user_runtime_scenarios>>). The next section (<<_detailed_sub_scenarios>>) describes aspects of the scenarios in more detail.

=== Generic Control Flow for write operations

The write flow branches depending on whether the uploaded content is RDF or non-RDF. RDF content goes through the full verification pipeline and is indexed in the graph database. Non-RDF content bypasses verification and graph storage entirely.

[mermaid, width=2000]
....
sequenceDiagram
    autonumber
    actor User
    participant Auth as Auth*
    participant API
    participant RdfDetector
    participant Verification
    participant AssetStore as AssetStore
    participant GraphDB as Graph-Database
    User ->> + Auth:Authenticate
    Auth -->> - User:OK
    User ->> + API:Upload content
    API ->> + RdfDetector:isRdf(contentType, content)
    RdfDetector -->> - API:true / false
    alt RDF content
        API ->> + Verification:Verify
        Verification -->> - API:OK
        API ->> + AssetStore:storeCredential
        AssetStore ->> + GraphDB:addClaims
        GraphDB -->> - AssetStore:OK
        AssetStore -->> - API:OK
    else Non-RDF content
        API ->> + AssetStore:storeUnverified
        Note right of AssetStore: No verification, no graph
        AssetStore -->> - API:OK
    end
    API -->> - User:OK
....

**Step 3** Participant's credential passed in request body in JSON-LD format

**Step 4** ParticipantsService.addParticipant is annotated as @Transactional

**Step 6** Return the Participant verification result. This should be an extension of API Participant structure.

**Step 7** The AssetStore must store credential content first in a FileStore, which is a non-transactional resource. In case of an exception on a subsequent step the file should be deleted in compensation action. Thus, the file identifier must be stored in some operation context.
The AssetMetadata store and GraphStore are transactional resources and must participate in the current operation transaction context.

**Step 11** Keycloak is a non-transactional resource, accessed via REST interface. In case of any error at group addition procedure, an exception must be thrown and the current transaction must be rolled back entirely.

Separately, a Participant-Administrator user for the new Participant is created. This should be aligned with the process in section "<<_adding_a_user>>". This process is initiated by the Portal.

==== Requesting a specific Asset

Assets can be retrieved by their IRI. The IRI is URL-encoded in the path parameter. The optional `?version=X` query parameter retrieves a specific historical version (1-based ordinal). Without it, the current version is returned. A non-existent version number returns HTTP 404.

[mermaid, width=2000]
....
sequenceDiagram
    autonumber
    actor User
    participant Auth as Authentication
    participant API
    participant AssetStore as AssetStore

    Auth -->> - User:OK
    User ->> + API:GET /assets/{id}[?version=X]

        AssetStore -->> - API: AssetMetadata (or 404)

        API ->> + AssetStore: getById(id)

    end
    API -->> - User:AssetMetadata

==== Adding a Credential for an Offering

The basic flow of creating and updating an offering is identical. The distinction cannot be made until the Asset Store receives the credential metadata and checks if it already exists.

The verification steps shown below are configuration-dependent. In the default profile, only syntax and semantic verification are active. In the strict profile, all steps (including schema validation and signature verification) are executed. See <<_verification_profiles>> for details.

[mermaid, width=2000]
....
sequenceDiagram
    autonumber
    actor User
    participant Auth
    participant API
    participant VerificationService
    participant AssetStore as AssetStore
    User ->> + Auth:Authorize(ASSET_CREATE | ADMIN_ALL)
    Auth -->> - User:[OK]
    User ->> + API:POST /assets
    API ->> + VerificationService:verifyOfferingCredential(credential)
    VerificationService --> VerificationService:[Verify Syntax]
    VerificationService --> VerificationService:[Verify Semantic]
    VerificationService --> VerificationService:[Extract Credential Type]
    opt schema validation enabled
        VerificationService --> VerificationService:[Verify Schema]
    end
    opt signature verification enabled
        VerificationService --> VerificationService:[Verify Signatures]
    end
    VerificationService --> VerificationService:[Extract Participant]
    VerificationService --> VerificationService:[Extract Claims]
    VerificationService -->> - API:CredentialVerificationResult
    API --> API:Check[credential-participant == user-participant | Ro-MU-CA]
    alt credential.type != Participant
        API -->> User: [Error. Use Participant API]
    else credential.type == Offering

==== Adding a Non-RDF Asset

Non-RDF assets (PDFs, plain JSON, YAML, images, binary files) are uploaded through the same `POST /assets` endpoint using `multipart/form-data` or `application/octet-stream`. The `RdfDetector` classifies the content; non-RDF assets bypass verification and graph storage entirely.

[mermaid, width=2000]
....
sequenceDiagram
    autonumber
    actor User
    participant Auth as Authentication
    participant API as AssetUploadController
    participant Service as AssetUploadService
    participant RdfDetector
    participant AssetStore as AssetStore
    participant FileStore as Asset FileStore

    User ->> + Auth:Authorize(ASSET_CREATE | ADMIN_ALL)
    Auth -->> - User:OK
    User ->> + API:POST /assets (multipart/form-data or application/octet-stream)
    API ->> + Service:processUpload(bytes, contentType, filename)
    Service ->> + RdfDetector:isRdf(contentType, bytes)
    RdfDetector -->> - Service:false
    Service ->> Service:computeSHA256(bytes)
    Service ->> Service:buildMetadata(hash, contentType, fileSize, participantId)
    Service ->> + IriGenerator:generateIri()
    IriGenerator -->> - Service:urn:uuid:... (UUID URN)
    Service ->> + AssetStore:storeUnverified(metadata, filename)
    AssetStore ->> AssetStore:insertMetadata(PostgreSQL)
    AssetStore ->> + FileStore:storeFile(hash, contentAccessor)
    FileStore -->> - AssetStore:OK
    AssetStore -->> - Service:storedMetadata
    Service -->> - API:AssetMetadata
    API -->> - User:201 Created (metadata with contentType, fileSize, iri)
....

**Step 5-6:** `RdfDetector` checks the MIME type against the configured allowlist and, for `application/json`, inspects content for a JSON-LD `@context` pattern.

**Step 7-8:** `IriGenerator` generates a UUID URN (`urn:uuid:...`) per RFC 4122 for the non-RDF asset. Each upload receives a unique IRI.

**Step 9:** `storeUnverified` persists the metadata (including the assigned IRI) in PostgreSQL. If an asset with the same content hash already exists, HTTP 409 (Conflict) is returned with the existing IRI in the error response.

**Step 11:** The raw bytes are written to the `assetFileStore`, not the schema/context File Store.

==== Updating an Asset

Assets are updated via `PUT /assets/{id}`. The update is applied in-place on the single row for that IRI; Hibernate Envers captures the previous state as a new revision in `assets_aud` automatically. No DEPRECATED row is created. The optional `changeComment` query parameter is recorded on the new revision.

The IRI extracted from the credential body must match the path `{id}`. If they diverge, HTTP 400 is returned.

[mermaid, width=2000]
....
sequenceDiagram
    autonumber
    actor User
    participant Auth as Authentication
    participant API
    participant VerificationService
    participant AssetStore as AssetStore
    User ->> + Auth:Authorize(ASSET_UPDATE | ADMIN_ALL)
    Auth -->> - User:OK
    User ->> + API:PUT /assets/{id}[?changeComment=...]
    API ->> API:Check[path id == credential body IRI — else 400]
    API ->> + VerificationService:verifyOfferingCredential(credential)
    VerificationService -->> - API:CredentialVerificationResult
    API ->> + AssetStore:storeCredential(AssetMetadata, CredentialVerificationResult)
    Note right of AssetStore: In-place UPDATE — Envers records<br/>previous state in assets_aud
    AssetStore -->> - API:OK
    API -->> - User:200 Asset
....

==== Query the catalogue

The query endpoint uses the `Content-Type` header to select the query language: `application/opencypher-query` for openCypher (Neo4j backend) or `application/sparql-query` for SPARQL-star (Fuseki backend). The request body is raw query text (not a JSON wrapper). Only read queries are allowed. Write queries (e.g., `CREATE`/`DELETE` in openCypher, `INSERT`/`DELETE` in SPARQL) are rejected with HTTP 500.

=== Description of shared procedures

==== Verify Credential

The verification of credentials consists of up to four stages, each independently configurable (see <<_verification_profiles>>).
Besides verifying the credential, additional tasks are executed (e.g., parsing, claim extraction).

`VerificationServiceImpl` acts as a thin Context in the Strategy pattern: it resolves per-request verification toggles and delegates all verification and claim extraction logic to a `VerificationStrategy` (currently `CredentialVerificationStrategy`). The strategy contains the full VP/VC parsing, semantic validation, schema validation, signature verification, claim extraction, and protected namespace filtering logic. If any triples referencing the `fcmeta:` namespace are filtered, a user-visible warning is included in the `CredentialVerificationResult`. See <<_verification_strategy_pattern>> for details.
**Syntactic validation** is always active. The credential is parsed as JSON-LD and validated against the Verifiable Credential data model. Any syntactic issue will raise an error here.

**Semantic verification** validates the credential subject `@type` against the loaded ontologies. This is active by default and catches type mismatches early.

**Schema validation ** (when enabled) checks the uploaded credentials against a composite schema of type SHACL.
If a credential does not conform to the composite schema, a verification exception is thrown with a message containing the validation report. In that case, the verification process does not proceed, and claims are not extracted.
Automatic schema validation against stored SHACL shapes has been **disabled in the upload verification flow**. Credentials are accepted regardless of their conformity to stored SHACL shapes. On-demand schema validation of stored assets against specific SHACL shapes is available through the dedicated `SchemaValidationService` (see <<_schema_validation_service>>).

NOTE: For backward compatibility, automatic schema validation during upload can be re-enabled via the configuration property `federated-catalogue.verification.schema=true`.

**Cryptographic verification (trust verification)** (when enabled) checks all proofs in the credential: whether the signature, the offered key, and the content match.
When checking whether the proofs match the credential or the contained Verifiable Credentials, the JSON object is normalized using the Universal RDF Dataset Canonicalization Algorithm URDNA2015footnote:[https://w3c-ccg.github.io/rdf-dataset-canonicalization/spec/#dfn-urdna2015[RDF Dataset Canonicalization. A Standard RDF Dataset Canonicalization Algorithm. W3C, Final Community Group Report 15 October 2022]] as specified by Gaia-X.
This allows robust signing of credentials.
The normalization algorithm extracts the RDF graph from the claims to perform the signing.
Thus, the credential can be changed (in some specific manners) while the signature remains.

When the Gaia-X Trust Framework is additionally enabled, the key is validated against the Gaia-X Trust Anchor Registry to verify a valid trust anchor chain via the `x5u` certificate URL. Without the trust framework, signature verification only proves internal consistency (that a DID controls a key) but cannot establish identity or trust — this is why signature verification is disabled in the default profile (see ADR 3).

Afterwards the claims are extracted from the credential.
Further, it is checked if all Credential Subjects in the credential affect the same subject.
